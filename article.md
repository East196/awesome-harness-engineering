# Anthropic 多智能体 Harness 设计：让 AI 持续开发复杂应用的工程实践

## 引言

在 AI 辅助编程领域，一个长期存在的挑战是：如何让 AI 代理在数小时的持续工作中保持高质量输出？Anthropic 的工程师 Prithvi Rajasekaran 最近分享了一套创新的 **Harness 设计模式**，通过多智能体架构解决了这个难题。本文将深入解析这套方法，并结合实际案例说明如何应用到你的项目中。

---

## 一、为什么简单的 AI 代理会"失控"

### 1.1 上下文窗口的诅咒

当 AI 代理执行长时间任务时，会遇到两个致命问题：

**上下文填满导致的失焦**
随着对话进行，模型会逐渐丢失对整体目标的把握。就像一个人连续工作几小时后注意力涣散一样，AI 也会在冗长的对话中"忘记"最初的目标。

**上下文焦虑（Context Anxiety）**
某些模型（如 Claude Sonnet 4.5）会在接近上下文限制时提前结束工作，即使没有完成任务。这是一种"自我保护"行为，但会严重影响产出质量。

![上下文管理对比](context-comparison.png)

*图：上下文焦虑问题与上下文重置解决方案对比*

### 1.2 自我评估的盲区

更隐蔽的问题是**自我评估偏差**：当要求 AI 评价自己的工作时，它往往过于宽容——即使输出质量平庸，也会自信地给出高分。这在主观任务（如 UI 设计）中尤为明显，因为没有像单元测试那样的客观验证标准。

> "Whether a layout feels polished or generic is a judgment call, and agents reliably skew positive when grading their own work."
> —— Anthropic Engineering Blog

## 二、核心创新：生成器-评估器循环（Generator-Evaluator Loop）

### 2.1 灵感来源：GAN 的对抗思想

Anthropic 从 **生成对抗网络（GAN）** 中获得灵感，设计了一个多智能体架构：

![多智能体 Harness 架构](diagram.png)

*图：多智能体 Harness 架构 — Planner、Generator、Evaluator 形成反馈循环*

### 2.2 三个智能体的分工

| 智能体 | 职责 | 关键能力 |
|--------|------|----------|
| **Planner** | 将简单需求扩展为完整产品规格 |  ambitious scope，聚焦产品上下文而非技术细节 |
| **Generator** | 逐个功能实现，自我评估后提交 | React/Vite/FastAPI/SQLite 技术栈，版本控制 |
| **Evaluator** | 通过 Playwright 实际测试应用，按标准评分 | UI 测试、API 测试、数据库状态验证 |

## 三、实战案例 1：前端设计质量提升

### 3.1 设计评估的四维标准

Anthropic 为前端设计制定了**可量化的美学标准**，将主观判断转化为可评分的指标：

![前端设计评估四维标准](design-criteria.png)

*图：将主观美学转化为可量化评分的设计评估标准*

**设计质量（Design Quality）**
- 设计是否像一个统一的整体，而非零件拼凑？
- 色彩、字体、布局、图像是否融合成独特的氛围和身份？

**原创性（Originality）**
- 是否有定制的创意决策，还是套用模板？
- 是否避免"AI 生成痕迹"（如紫色渐变配白卡片）？

**工艺（Craft）**
- 技术执行：排版层级、间距一致性、色彩和谐、对比度
- 基础能力检查，而非创意检查

**功能性（Functionality）**
- 用户能否理解界面功能？
- 能否找到主要操作并完成任务？

### 3.2 迭代过程的惊人发现

在实际运行中，评估器会**实际浏览页面**（通过 Playwright MCP），截图并仔细研究后再给出评分。每次迭代通常需要 5-15 轮，完整运行可达 4 小时。

**关键发现：**

1. **评分与偏好的非线性关系**
   评分通常随迭代提升，但作者经常更喜欢中间某版而非最终版。这说明自动评分需要人工最终把关。

2. **创意跳跃现象**
   在一个荷兰艺术博物馆网站的案例中，第 9 迭代还是传统的深色落地页，第 10 迭代突然转变为**空间体验**：CSS 3D 透视渲染的棋盘地板、自由悬挂的艺术品、门洞导航——这种创意飞跃在单次生成中从未出现。

3. **提示词对风格的塑造**
   像"最佳设计应达到博物馆品质"这样的措辞，会推动设计向特定视觉风格收敛。提示词本身就在塑造输出特征。

## 四、实战案例 2：全栈应用开发

### 4.1 从设计到代码的迁移

将前端设计的经验应用到全栈开发，核心架构保持一致：

```
┌────────────────────────────────────────────────────────────┐
│                  全栈开发 Harness 流程                      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. Planner 接收 1-4 句简单描述                              │
│     ↓                                                      │
│  2. 扩展为完整产品规格（包含 AI 功能集成点）                   │
│     ↓                                                      │
│  3. Generator 按 Sprint 逐个实现功能                         │
│     - React + Vite + FastAPI + SQLite/PostgreSQL           │
│     - 每个 Sprint 结束自我评估                               │
│     ↓                                                      │
│  4. Evaluator 实际测试运行中的应用                            │
│     - Playwright 点击测试                                    │
│     - UI/API/数据库状态验证                                  │
│     - 按标准评分，未达标则反馈给 Generator                    │
│     ↓                                                      │
│  5. 循环直到所有 Sprint 完成或达到质量阈值                     │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 4.2 关键技术决策

**上下文重置 vs 自动压缩**

早期版本使用 Sonnet 4.5，存在"上下文焦虑"问题，必须在会话间重置上下文。新版本使用 Opus 4.5 后，这一问题基本消失，可以**单会话连续运行**，依靠 Agent SDK 的自动压缩管理上下文增长。

**技术栈选择**
- 前端：React + Vite
- 后端：FastAPI
- 数据库：SQLite（早期）→ PostgreSQL（后期）
- 版本控制：Git

## 五、关键经验总结

### 5.1 什么有效

| 策略 | 效果 | 适用场景 |
|------|------|----------|
| **生成器-评估器分离** | 显著提升质量 | 主观任务（设计）、复杂任务（全栈开发） |
| **可量化的评估标准** | 将主观判断转化为可优化目标 | 设计质量、代码质量 |
| **上下文重置** | 解决上下文焦虑和失焦 | 使用 Sonnet 4.5 等早期模型 |
| **单会话+自动压缩** | 降低编排复杂度 | 使用 Opus 4.5 等新模型 |
| **实际交互测试** | 发现静态检查遗漏的 bug | 全栈应用评估 |

### 5.2 什么无效或需谨慎

- **自我评估的乐观偏差**：必须分离生成和评估角色
- **评分的绝对化**：高分不等于更好，人工审核仍必要
- **复杂度递增**：后期迭代往往更复杂，需权衡维护成本

## 六、如何应用到你的项目

### 6.1 起步建议

1. **从简单评估标准开始**
   不必追求完美标准，先定义 3-5 个可量化的维度

2. **使用 Playwright MCP 进行实际测试**
   静态分析无法替代真实用户交互

3. **设计反馈循环**
   确保评估器的输出能被生成器理解并执行

4. **监控迭代质量曲线**
   如果评分不再提升，考虑停止或更换策略

### 6.2 工具推荐

- **Claude Agent SDK**：提供 Agent 编排和上下文管理
- **Playwright MCP**：让 AI 能够实际浏览和测试应用
- **Git**：版本控制是回溯和分支的基础
